# Sprint-01 Task #3 - MBTI 人格傾向明確影響行動機率 (CLINE integrator)

> 對應 sprint: `projects/ProjectLivingWorldSim/sprints/sprint-01.md` 任務 #3
> 前置交付: `projects/ProjectLivingWorldSim/app/`
> 本檔自包含。CLINE 用自己的 PLAN 模式決定實作細節；本檔只給 input、output、驗收條件與硬性技術約束。

# INPUT

## 1. Sprint 任務描述

任務：MBTI 人格傾向明確影響行動機率

完成度判定方式：

- 角色的 I/E、S/N、T/F、J/P 數值會明確影響移動、活動、物件行動、交談、共同活動、待機等候選行動權重。
- 事件詳情或候選權重摘要會顯示 MBTI 修正來源。
- 角色編輯器能讓玩家理解各人格軸會影響哪些行動傾向。
- 反向：人格數值缺失或舊存檔人格資料不完整時不得造成模擬崩潰。
- 反向：所有人格修正不得直接禁止任何行動。

## 2. 使用者回饋

使用者希望「機率」與 MBTI 有關。

目前實作已經有部分 MBTI 影響：

- E/I 影響移動、交談、待機。
- S/N 影響物件行動。
- T/F 影響交談。
- J/P 影響活動與移動。

但目前問題是：

- 玩家不容易看出 MBTI 正在影響機率。
- 權重摘要裡只有「個性修正」這種概括文字，不足以理解是哪個人格軸造成。
- 角色編輯器沒有說明人格軸會影響哪些行動傾向。

本任務不是新增完整心理模型，而是把既有 MBTI 光譜與行動權重的關係變得明確、可讀、可測。

## 3. 設計語意

人格數值只影響候選行動權重，不直接禁止行動。

例如：

```text
I 高的角色仍然可以主動交談，只是交談權重較低、待機或獨自行動權重較高。
E 高的角色仍然可以獨處，只是交談、共同活動、移動權重較高。
```

MBTI 標籤仍只是顯示結果；實際計算使用 8 個數值：

- I：內向
- E：外向
- S：實感
- N：直覺
- T：思考
- F：情感
- J：判斷
- P：感知

## 4. 必要行為映射

CLINE 可以調整倍率，但必須讓每個人格軸至少明確影響以下行為。

### I / E

I 較高時提高：

- 待機
- 獨自活動
- 低社交行動

E 較高時提高：

- 移動到其他地點
- 主動交談
- 共同活動

### S / N

S 較高時提高：

- 使用具體物件
- 參與偏實作、偏現場的活動

N 較高時提高：

- 抽象或興趣話題
- 活動中的話題觸發
- 探索性移動或非日常行動

第一版如果活動/話題尚無「抽象/具體」分類，可先讓 N 影響「話題相關行動」權重，S 影響「物件行動」權重。

### T / F

T 較高時提高：

- 中性或理性話題
- 低情緒成本的行動

F 較高時提高：

- 交談
- 共同活動
- 正面或負面情緒話題
- 關係導向行動

第一版如果話題只有 `moodTone`，可讓：

- T 偏好 `neutral`
- F 偏好 `positive` / `negative`

### J / P

J 較高時提高：

- 地點內既有活動
- 穩定、可預期的活動
- 延續目前活動

P 較高時提高：

- 移動
- 嘗試物件行動
- 切換活動
- 隨機探索

## 5. 權重摘要需求

事件詳情目前已有「權重摘要」。本任務需要把人格修正拆得更明確。

不要只顯示：

```text
個性修正 +8.5
```

應顯示類似：

```text
MBTI 修正：E +6.0、F +3.0、J -2.0
```

或：

```text
人格影響：E 提高交談 +6.0，F 提高關係行動 +3.0，I 降低社交 -4.0
```

最小可接受：

- 每個候選行動的 `weightSummary` 必須列出具體人格軸，而不是只寫「個性修正」。
- 摘要應來自 engine/action resolver 的實際計算，不要 UI 硬寫。

## 6. 角色編輯器 UI 需求

角色編輯器的人格區需要增加簡短說明，讓玩家知道人格會影響機率。

可用形式：

```text
人格會影響行動權重，不會直接禁止行動。
I/E：獨處 vs 社交與移動
S/N：物件行動 vs 話題/探索
T/F：理性話題 vs 關係互動
J/P：固定活動 vs 隨機探索
```

不要做成長篇教學頁；放在角色人格區附近即可。

## 7. 舊資料與防呆

人格資料可能來自舊存檔或手動 import，必須防呆：

- 缺少某個人格值時，以 50 補齊。
- 人格值超出 0-100 時 clamp。
- `temporaryModifiers` 缺失時視為空陣列。
- 計算有效人格時不得產生 `NaN`。

如果目前已有部分防呆，請補測試確認。

## 8. 測試要求

需要新增或更新測試，至少覆蓋：

- E 高的角色交談或共同活動候選權重高於 I 高角色。
- I 高的角色待機權重高於 E 高角色。
- S 高的角色物件行動權重高於 N 高角色。
- J 高的角色活動候選權重高於 P 高角色。
- P 高的角色移動候選權重高於 J 高角色。
- T/F 對話題或交談權重產生可測差異。
- 人格缺欄位時不崩潰，且使用 50 缺省。
- `weightSummary` 包含具體人格軸文字，例如 `I`、`E`、`S`、`N`、`T`、`F`、`J`、`P` 中相關者。

測試不需要依賴隨機分布；可直接檢查候選 action 的 weight 或 weightSummary。

## 11. 驗收條件

### 正向

- 玩家調高 E 後，更容易看到交談、共同活動或移動相關權重提高。
- 玩家調高 I 後，更容易看到待機或低社交行動權重提高。
- 玩家調高 S 後，物件行動權重提高。
- 玩家調高 N 後，話題或探索傾向提高。
- 玩家調高 F 後，交談或關係導向行動權重提高。
- 玩家調高 T 後，中性/理性話題或低情緒行動權重提高。
- 玩家調高 J 後，地點既有活動權重提高。
- 玩家調高 P 後，移動、切換或探索行動權重提高。
- 事件詳情的權重摘要可以看出 MBTI 具體影響來源。
- 角色編輯器能讓玩家理解人格值會影響哪些行動機率。
- `npm test` 通過。
- `npm run build` 通過。

### 反向

- 人格值缺失時不崩潰。
- 人格值超出 0-100 時不造成 NaN。
- 人格修正不得把任何行動權重壓到 0 以下後完全消失；仍需保留最低可選權重。
- 沒有活動、物件或互動對象時，仍能產生待機事件。
- 舊存檔 import 後仍可 tick。

## 12. 硬性約束

- 人格機率邏輯必須在 engine/action resolver 或可測試 helper 中，不得只存在 UI。
- `weightSummary` 必須反映實際計算。
- 不得把 MBTI 四字母標籤當成計算來源；必須使用 8 個實際數值。
- 不得讓人格變成硬限制。
- 不得破壞現有地點/活動/話題基礎權重。
- 不得改動 git 狀態。

- 每個人格軸如何影響機率。
